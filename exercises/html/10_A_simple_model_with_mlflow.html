Experiment Tracking und Model Registry

In this exercise we extend the code that trains our simple model with Experiment Tracking. We conduct several experiments and compare their results in MLFlow. We add metadata to our models, register a model in the Registry and briefly try out the MLFlow Client (the low-level API of MLFlow).

Training and evaluation

Now it's time to evaluate the model. To do this, we need to determine which metrics we use (actually, this is a very central step that must be performed at the very beginning of an ML project, because the metrics with which we evaluate the model must be as well aligned with the business goal).
We can log the classic accuracy, precision, recall and F1-score here. Import these metrics from sklearn.metrics
Suggested solution 
from sklearn.metrics import accuracy_score, precision_score, recall_score, f1_score
We validate with a simple, fixed validation set. Use train_test_split() once more on the training set to split off a fixed validation set from it.
Suggested solution 
X_train_small, X_val, y_train_small, y_val = train_test_split(X_train, y_train, random_state=42)
With this we can now train with the _small variants (X_train_small, y_train_small) and validate with _val. Up to and including this step there is no MLFlow-specific code.
Use fit() to train and y_pred = predict() to make predictions for the subsequent validation.
Suggested solution 
clf.fit(X_train_small, y_train_small)
y_pred = clf.predict(X_val)
Now we fill a dictionary with our four metrics. 
    metrics_static_val_set = {
        "accuracy": accuracy_score(y_val, y_pred),
        "precision": precision_score(y_val, y_pred),
        "recall": recall_score(y_val, y_pred),
        "f1": f1_score(y_val, y_pred),
}
And create an MLFlow signature for the training data.
Suggested solution 
from mlflow.models import infer_signature
signature = infer_signature(model_input=X_train, model_output=y_train)
And now that we have signature, parameters, model and scores, we can start the MLFlow run and log all four things.
As a reminder: To avoid logging a half-finished run when an exception occurs during training, we only start the MLFlow run for logging when we have all components for logging together, i.e. only after training and evaluation.
Give the run a name. Which name you give the run depends on what you are currently experimenting with, which hyperparameters you want to test, etc.
Suggested solution 
with mlflow.start_run(run_name="baseline model") as run:
    mlflow.log_params(params)
    mlflow.log_metrics(metrics_static_val_set)
    mlflow.sklearn.log_model(
            sk_model=clf, signature=signature, name="mushroom"
    )

    # This call should execute quickly
    # If it takes more than a few seconds, the problem is most
    # likely that the top-level artifact bucket has not been created.
    # You would see this also in the docker log output of the mlflow service
    # If you have this problem, go back to the end of the previous exercise
    # where you create the bucket.
If everything worked, we see in the MLFlow UI (you cannot use the URLs output in the notebook, you must open the Model Registry via codespaces under PORTS) that metadata, the four metrics, the parameter(s) and the model were logged. When we click on a metric, it is displayed as a plot. The Y-axis corresponds here to the value of the metric, the X-axis to so-called steps. Steps can be used to map training over time, i.e. a learning curve over epochs, over training set sizes or other hyperparameters. If we don't specify the step= argument in log_metrics(), by default only one step is logged, which then leads to the constant plot as we see it here.

K-Fold Cross Validation

Evaluation on a fixed validation set is only recommended when we have a lot of data, and our validation set is large enough and thus representative of the entire training set. Smaller datasets and fast-training models allow for K-fold cross validation and thereby more precise statements about our model performance. How would we log results of a K-fold cross validation in MLFlow?
Unfortunately, MLFlow does not support this directly. As a workaround, we could log the average over our folds.

Metadata

Runs can be provided with a description and with tags. This can either happen during logging of the run, or afterwards in the UI or via the API (described further below).
Add a description and a new tag to a run using the UI.
Suggested solution
This can be done in the Overview tab of a run.
You can also add metadata to the model itself. These then appear in the MLmodel file.
Log a new run and add model metadata. Find this metadata in the UI.
Suggested solution 
# you need to add a metadata parameter with a dictionary when calling log_model()
# you can then see the metadata in the MLmodel File
with mlflow.start_run() as run:
    mlflow.sklearn.log_model(
        sk_model=clf, signature=signature, name="mushroom", metadata={'foobar':'baz'}
    )

Register models

So far we have only been in the development phase. Once we have reached the point where we have a model that we want to deploy, we must register it.
We can do this either in the MLFlow UI by clicking on the desired model in the run overview in the Models column, then a Register model button appears in the top right. Or we can do it directly from the log_model() call, using the parameter registered_model_name="some model name". Or we do it with the function mlflow.register_model() or via API (see below).
If we choose a model name that does not yet exist, a new model in version 1.0 is registered. If we choose a name that already exists, the model is registered under this name and the version is incremented by one. Try this out once.
Registered models appear under the Models tab. A model has a name, a version and tags. Versions can have an alias, which are explained further below.

MLFlow Client API

So far we have worked with the higher-level mlflow module. But there is also a second API that allows low-level access to MLFlow, the MLFlow Client, which provides a simple CRUD interface for experiments, runs and models.
Use the MLFlow Client to list all registered models.
Suggested solution 
from mlflow import MlflowClient

client = MlflowClient()
client.search_registered_models()
Retrieve a model using get_model_version() and output its description, alias and version.
Suggested solution 
model = client.get_model_version(name='name of one of your models', version="1")
model.description, model.aliases, model.version
Add a description to your model using update_model_version(). The description is attached to the version. There is also a description that is attached to the model. Check in the UI afterwards whether this worked.
Suggested solution 
client.update_model_version(name='name of one of your models', version="1", description="foobar")

Model version aliases

Models can be loaded either via path, via name+version or via alias, see the documentation.
Model version aliases are identifiers that are attached to a version. They are used to decouple the calling code from the model version. We can give our active, deployed model the alias champion (as in "our currently best-perorming model") and reference it via this. If we develop a new, improved model, we can assign the alias champion to this new model and don't need to adjust the calling code.
Register your best model, if not already done, then give it the alias champion using the function set_registered_model_alias(). You will use this model later.
Suggested solution 
client.set_registered_model_alias(name='name of one of your models', version="1", alias='champion')
